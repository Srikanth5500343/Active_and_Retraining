RackTrack · Deployment

Two options to ship the APK

Your server is on 24/7 already. So the choice is simple — keep it on your own machine and front it with Cloudflare, or move the heavy parts to AWS. This is what each one means concretely, what it costs, and what to do.

2 The one rule

Netdisco and the TP-Link SSH probe must run on your office LAN. Cloud servers can't reach 192.168.x.x. This is why both options below keep something on-prem.

Everything else (login, scan, ML pipeline, ServiceNow, PDFs, notifications) works fine from anywhere with internet.

3 Building the APK

Same for both options. The only thing that changes is the URL you bake in.

cd client
npm ci
VITE_API_BASE=https://your-domain.com npm run build
npx cap sync android
cd android
./gradlew assembleDebug          # unsigned, sideload
# OR
./gradlew assembleRelease        # signed, for Play Store

The APK only carries one config: VITE_API_BASE. No secrets, no keys, no server code. Build once with the right URL and the APK keeps working as long as that URL keeps working.

Where to actually run the build:

  • Locally — install JDK 17 + Android SDK 34, run the commands above.
  • GitHub Actions — workflow that runs on push, uploads APK as an artifact. Free for private repos up to 2,000 min/mo.

A Cloudflare on your 24/7 server Recommended for now

Hardware: $0 (you have it)  •  Domain: $12 / year  •  Cloudflare: $0 / month  •  Power: ~$2 / month

The picture

┌─────────┐ HTTPS to your domain │ Phone │ ──────────────────────────────┐ │ (APK) │ │ └─────────┘ ▼ ┌─────────────────────┐ │ Cloudflare edge │ │ (free TLS + DNS) │ └──────────┬──────────┘ │ outbound tunnel │ (no port-forwarding) ▼ ┌──────────────────────────────────────────────────┐ │ YOUR 24/7 SERVER (the one you already have) │ │ ───────────────────────────────────────── │ │ cloudflared (named tunnel, runs as a service) │ │ Node server :3001 │ │ Python pipeline + Models/ │ │ Docker: netdisco stack (web on :5000) │ │ SSH probe → switches on the same LAN │ │ Encrypted creds in server/.env + .env.key │ └──────────────────────────────────────────────────┘

What you need to set up

  1. Buy a domain (~$12/year). Cloudflare Registrar, Porkbun, Namecheap — pick one.
  2. Add the domain to a free Cloudflare account. Cloudflare gives you nameservers; set them at the registrar.
  3. Install cloudflared on your 24/7 server: 
    # Linux
curl -L --output cloudflared.deb https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64.deb
sudo dpkg -i cloudflared.deb

# Windows
winget install --id Cloudflare.cloudflared
  4. Authenticate, create a named tunnel, point DNS at it: 
    cloudflared tunnel login
cloudflared tunnel create racktrack
cloudflared tunnel route dns racktrack racktrack.yourdomain.com
  5. Create ~/.cloudflared/config.yml: 
    tunnel: racktrack
credentials-file: /home/you/.cloudflared/<UUID>.json
ingress:
  - hostname: racktrack.yourdomain.com
    service: http://localhost:3001
  - service: http_status:404
  6. Install as a service so it survives reboots: 
    sudo cloudflared service install
sudo systemctl enable --now cloudflared
  7. Build the APK with the new URL: VITE_API_BASE=https://racktrack.yourdomain.com.

Good things

  • Cheapest by a wide margin. ~$12/year in real spend.
  • Stable URL. Survives restarts forever — unlike the trycloudflare.com URL you have today.
  • Free TLS, free DNS, free DDoS protection from Cloudflare.
  • No port-forwarding. The tunnel makes outbound connections to Cloudflare, so you don't open anything on your home/office router.
  • Network View and Available Ports work natively — the server is on the same LAN as the switch and Netdisco, no VPN bridge needed.
  • One machine, one place to log in. Easy mental model.

Limitations

  • Single point of failure. If your machine fails, dies, or your ISP drops, the app goes down. No automatic failover.
  • Backups are your job. The SQLite auth DB at server/data/, the encrypted creds, and scan outputs all need a backup plan (rsync to a USB drive / cloud storage).
  • ~100 MB upload limit per HTTP request on free Cloudflare. Your scan photos are well under this — fine.
  • No SLA. Cloudflare's free tier is very stable in practice but doesn't guarantee uptime.
  • Bandwidth is your home/office uplink. If lots of techs scan at the same time, your upload speed becomes the bottleneck.
  • Maintenance is on you. OS patches, Node/Python updates, Docker images, log rotation.

The Node + Python work moves to AWS. Your existing 24/7 server sticks around but only runs Netdisco + Tailscale, because of the LAN-reach rule. Tailscale links the two so the EC2 can talk to Netdisco at http://<tailscale-ip>:5000 and SSH into your switches at their normal LAN IPs.

Set-up steps (high level)

  1. Launch t3.medium, Ubuntu 22.04, 30 GB EBS, security group allowing SSH (your IP) + 80 + 443.
  2. Buy a domain, add to Route 53, point an A record at the EC2's elastic IP.
  3. On the EC2: install Node 20, Python 3.10, Docker, Chromium (for puppeteer), Caddy, Tailscale.
  4. git clone your repo. Pull the 400 MB Models/ from S3 (or commit via Git LFS).
  5. Generate a fresh server/.env.key on the EC2 (don't reuse dev key). Re-run node server/encrypt-creds.js set tp-link with your switch creds.
  6. Install Tailscale on EC2 and on your on-prem 24/7 server. Both join the same tailnet.
  7. On the on-prem box, advertise the LAN subnet so EC2 can reach the switch directly: 
    sudo tailscale up --advertise-routes=192.168.0.0/24
    Approve the route in the Tailscale admin console.
  8. On the EC2, in server/.env: NETDISCO_URL=http://<onprem-tailscale-ip>:5000.
  9. Caddyfile: 
    racktrack.yourdomain.com {
    reverse_proxy 127.0.0.1:3001
    encode gzip
    request_body { max_size 50MB }
}
  10. systemd unit for the Node server, auto-start on boot.
  11. Build APK with VITE_API_BASE=https://racktrack.yourdomain.com.

Good things

  • Real uptime. EC2 stays up through your office power cuts and ISP issues.
  • Independent of your office internet for everything except Network View / Ports.
  • Easy to scale up. Bump to t3.large ($36/mo reserved) in 30 seconds if scans queue.
  • S3 + EBS snapshots = backups handled.
  • Secrets Manager is the right place for the AES master key.
  • HTTPS via Caddy = no manual certificate management.

Limitations

  • Costs ~$25/mo minimum, indefinitely.
  • Still need the on-prem box. AWS doesn't solve LAN reach. Netdisco + the switch SSH probe stay where they are.
  • You manage the server. OS patches, security updates, log rotation, monitoring are on you.
  • Two places to debug. When something breaks, "is it AWS, on-prem, or Tailscale?" becomes a triage question.
  • RAM ceiling. 4 GB on t3.medium handles the ML pipeline fine for one or two parallel scans. If you have 5+ techs scanning simultaneously, jump to t3.large.
  • Egress costs kick in past 100 GB/mo (free tier) at $0.09/GB.
  • Models/ folder. 400 MB has to live somewhere — Git LFS (free up to 1 GB), or upload to S3 and pull at deploy time.

Recommendation: Start with Cloudflare on your server. It's $36/year all-in, gives you a stable URL, and uses the machine you already keep running. If it ever stops being enough, your 24/7 server becomes the bridge box in the AWS setup — same hardware, just a smaller role. No rework, no rewrites.

Right before you flip the switch

  1. Generate a fresh server/.env.key on the production host (don't reuse dev).
  2. Re-run node server/encrypt-creds.js set tp-link with your real switch creds — they're encrypted under the new key.
  3. Set NETDISCO_URL, NETDISCO_USER, NETDISCO_PASSWORD in server/.env.
  4. Set SN_INSTANCE, SN_USER, SN_PASSWORD for ServiceNow.
  5. Verify https://<your-domain>/api/health returns 200.
  6. Verify https://<your-domain>/api/netdisco/health returns OK.
  7. Build APK with the right URL, sideload, run a full scan.
  8. Distribute.
