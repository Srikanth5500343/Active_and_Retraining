Everything logs now

Live
Production had ~70 scattered console.logs, zero metrics, and silent crashes. Now every request, operation, and error flows through one structured logging + metrics pipeline.
⚠️

What was wrong

"Scans are slow" → nothing to point at. No latency histogram. No error rate. Process crashes left no log. SOC2 audit blocker because audit trail wasn't ingestable into log aggregators.

🔧

What we did

One module: pino for structured logs, prom-client for metrics, request-ID per request, span timer for operations, audit mirror, process-level safety net. Swept all 70 console.* calls.

⭐

Why it matters

You can finally diagnose production. Grep one request-ID through the entire stack. See p95 scan latency in Grafana. SOC2 / ISO 27001 ready. Server can no longer crash silently.

📋
Structured JSON logs
Every line carries service / env / version / requestId / userId. Secrets auto-redacted (passwords, tokens, cookies).
📊
Prometheus metrics
HTTP duration histogram, error counters by route, business-event counters, span timings, worker lifecycle.
⏱️
Span tracing
withSpan(name, fn) wraps any async op — times it, logs start+end, records duration histogram, counts errors.
🛡️
Crash safety net
Express error handler + process-level handlers. Unhandled errors get a fatal log + counter before exit. No silent deaths.
🔍
Audit log mirror
Every audit-log row also emits a structured pino line, so SOC tooling sees the same trail as the SQLite DB.
❤️
/healthz + /metrics
Health probe for k8s / load-balancers. Metrics endpoint for Prometheus / Grafana / Datadog.

Self-improving ML models

Live
Models were static — once trained, they never learned from real customer corrections. Now there's a closed loop: corrections in → cache → automatic retrain → manual promote.
⚠️

What was wrong

Every correction a user made was thrown away. Model accuracy was frozen on day one. Competitors who close this loop compound accuracy quarter over quarter; we couldn't.

🔧

What we did

Built a per-model staging cache. When N corrections accumulate, runner exports + trains. New models become versioned candidates kept forever. Operator manually promotes after evaluation — production never auto-replaced.

⭐

Why it matters

Every customer scan is now free training data. The model gets smarter with usage. Plus: every old version kept, easy A/B tests, easy rollback. Risk-free.

Multi-tenancy — each company in its own space

Live
Before: every signup landed in one shared bucket. Two companies on the same server saw each other's racks. Now each company has its own private space.
⚠️

What was wrong

If Acme and Globex both signed up, Acme's user could see Globex's racks. Cannot sell hosted SaaS without fixing this. SOC2 / GDPR audits would hard-fail.

🔧

What we did

Added a tenants table. Signup requires Company name → creates a tenant. JWT carries tenantId. Middleware filters every rack-touching route by tenant ownership. 404 (not 403) on cross-tenant — no info leak.

⭐

Why it matters

Hosted SaaS is now possible. Single deploy, multiple customers, guaranteed isolation. Audit-ready. Easy to do today; quarter-long retrofit if delayed.

  • Signup form requires Company name — every new user creates a fresh tenant they're founding member of.
  • JWT carries tenantId — login token already knows which company you belong to.
  • Global :rackId guard — every endpoint that takes a rack ID auto-filters by tenant ownership.
  • Same image scanned by 2 tenants → both co-own the rack ID. Neither leaks to the other.
  • Audit log carries tenant ID — admin queries are tenant-scoped (no cross-tenant snooping even for admins by default).
  • Feedback endpoints guarded too — even endpoints that take rack ID in the body (not URL) are checked.

⚠️

What was wrong

iOS is half the enterprise install base. Capacitor config targeted only Android. No AR feature — competitors could and would replicate any non-AR feature within a quarter.

🔧

What we did

Scaffolded full iOS Xcode project. Built custom Capacitor AR plugin: ARKit on iOS (fully working), ARCore plugin glue on Android (GL renderer is documented stub). New /ar-scan page in React.

⭐

Why it matters

Demo-winning feature. "Point phone at rack → device names overlay live → tap to see ports" is the union of three things you already have (CV + ports + camera). Competitors take 6+ months to copy.

How the AR overlay works

  1. iOS plugin opens a fullscreen ARKit view over the WebView
  2. Camera frames stream to JS at 1Hz (throttled, single-flight to backend)
  3. JS posts the frame to existing /api/analyze → gets device list back
  4. Plugin raycasts each bounding-box center against the tracked vertical plane → anchors a billboarded label there
  5. Tap a label → React navigates to that device's ports page
Status: iOS is fully functional Swift. Android plugin glue + activity scaffold are complete; the ARCore GL renderer is a 1-2 day stub that needs a physical Android device to finish.

Multi-rack video scanning

Live
User pans a phone video across N racks side by side. The system automatically splits the video, detects each rack, and produces a separate full report per rack — fully automated, no taps required.
⚠️

What was wrong

One video → one rack scan. A field tech with a 20-rack room had to take 20 separate scans. Painful. Real DCIM tools handle whole rooms.

🔧

What we did

Auto-split: sample frames, detect devices, watch where horizontal cluster center "jumps" — that's a new rack. Pick best frame per rack, run the existing analyze on each. Returns a group with N member rack IDs.

⭐

Why it matters

One pan, three reports. 20× the throughput for whole-room scanning. Competitive with DCIM tools (Sunbird / Nlyte / Device42) on this dimension. Each rack still a regular scan — Ports / Topology / SFP all work as-is.
