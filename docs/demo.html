Rack-scan App — Feature Reference

One page. Every feature in plain English with enough technical depth to explain what it does, how it works, and which trade-offs were chosen. Each section has an authorship pill so you know which parts your team built, which parts were built in our sessions together, and which are still pending.

00

What the app actually does

Team

A field engineer takes a photo of an open rack. The app figures out what's inside it — which switches, which ports are free, what cables go where, whether the firmware is up to date. The same data syncs to ServiceNow CMDB so the office team has the same picture without anyone manually retyping serial numbers.

The user journey

  1. Open the app, log in.
  2. Tap Scan Rack, point the camera, take a photo.
  3. Server runs CV → OCR → topology generation. The phone also fires an SSH probe of the live switch and a CMDB lookup in parallel.
  4. Land on the Results page — every tab (Overview / Ports / Topology / Network / Switches) is already populated. No per-tab spinner because everything fired the moment the scan finished.
  5. If something didn't get picked up automatically, the user types the missing make / model / firmware. Manual entries flow into the same vendor lookups as auto-detected values.
  6. From Results, sync to CMDB (creates a ServiceNow change ticket) or share by Slack / Teams / Outlook.

02

Rack scan — photo to inventory

Team

The user uploads a rack photo. The server runs a quality check, then computer vision detects each chassis, then OCR reads each chassis's labels. By the time the user lands on Results, every detected switch has its make / model / firmware filled in (or flagged as needing manual entry).

What you'd say in a demo

Quick demo line "We don't ask the user to type anything that we can read off the photo. They take one shot, we find every device, classify it, read the chassis sticker. The whole thing finishes in 20-25 seconds on cache miss, instant on cache hit."

How it works internally

  • Photo-quality gate runs first — blur, tilt, lighting, framing. Bad photo? User sees "retake straight-on" with a Proceed-anyway override.
  • CV pipeline — YOLO on the rack image finds device bboxes, dual-model approach (general + server-class) with low-confidence retry catches unusual chassis. Maps each box to a U-position.
  • OCR pipeline — for each switch/server/router/firewall, crops its bbox and runs three OCR passes (raw, CLAHE, unsharp), merges results, runs the parser.
  • Per-rack folder on disk: outputs/<rackId>/ holds the original image, the device map JSON, the OCR results, the topology JSON, and any CMDB ticket state.
  • RackId = SHA-256 prefix of the normalized image. Same photo → same rackId → cache hit on re-upload.

Key files

  • server/app.js:1241 — /api/analyze route
  • pipeline/runner.py — CV pipeline
  • pipeline/ocr_devices.py — per-bbox OCR + parser
  • client/src/pages/ScanPage.jsx — upload UI

Quality check — the photo gate

Team

Before the heavy CV pipeline runs, a fast sanity check looks at the image: is it sharp, straight, well-lit, framed? If not, the user gets a specific reason ("photo's tilted — retake straight-on") with a Retake or Proceed-anyway button.

The whole point: if the photo's bad, fail in 1.5 seconds with a clear explanation, instead of letting the user wait 30 seconds for a worthless result.

Heuristics

CheckMethodTrigger
BlurLaplacian variance"Photo is too blurry"
TiltHough lines, angle of dominant line"Photo's tilted"
Brightnessmean luma"Lighting issue"
Glarelocal variance map, saturated patches"Glare detected"
Framingedge count + aspect"Make sure full rack fits"
Resolutionmin(W, H)"Photo is too small"

After CV runs, a second check rejects "0 devices found" or "<3 U-rails detected" as not-a-rack-photo cases. Both bypass-able with the Proceed-anyway override.

Key files

  • pipeline/quality_check.py — the actual checks
  • server/worker-pool.js — keeps a Python worker warm so this finishes in ~1.5s
  • server/app.js:1271, 1320 — call sites + post-pipeline gate
04

CV detection — finding devices

Team

Given a rack photo, the CV pipeline draws boxes around each device (switch, server, patch panel, UPS, blank panel) and labels each box with a class name and a U-position.

Models used

ModelJobFile
Devices (general)YOLO; switch/router/patch-panel/PDU/UPS/empty/etc.Models/best 32.pt
Devices (server)YOLO trained for servers + blade enclosuresModels/best 33.pt
UnitsYOLO finds U-rail markersModels/unit.pt
Port countYOLO on chassis crops, counts portsModels/port_count.pt
Cable classifierEfficientNet; CAT/fiber/DACModels/best_model_efficientnet.pth

Two-stage detection

  • Primary pass: dual-model at conf=0.20-0.25.
  • Low-confidence retry: same models at conf=0.08, IoU-deduped against primary. Catches unusual chassis (CRS518 wavy fascia, NETGEAR PoE, unusual cable channels) the primary missed.

U-position mapping

Run the unit model to get U-rail tick boxes (U01, U02, …). For each device bbox, find the U-numbers whose Y-range overlaps. A 2U server might be tagged ["U10","U11"]; a 1U switch ["U10"].

Honest caveat On the 12-image test set, CV emits 157 device bboxes but only ~30 are real switches (per manual ground truth). The OCR class filter narrows the focus, but over-detection is a real cost — see Known limits.

Key files

  • pipeline/runner.py — pipeline entry
  • pipeline/detection.py — YOLO loading + dual-model dedup + Hough rack-bounds
  • config.json — confidence thresholds + model paths
05

OCR — reading text from chassis

Shared

Per-bbox OCR. For each detected switch/router/server, crop the chassis from the photo, run text recognition on just that crop, parse what came back into vendor / model / firmware fields.

Three preprocessing passes

  1. Raw crop — works for cleanly-lit, sharp text.
  2. CLAHE contrast-enhanced — works for faded labels in shadow.
  3. Unsharp mask — works for cleanly-lit chassis where CLAHE adds noise.

Each pass produces text + confidence. We merge, dedupe by lowercased text, keep the best confidence per phrase. Different preprocessing catches text any single pass misses.

The parser (two-tier)

  1. Model regex — ~24 patterns covering Cisco / TP-Link / D-Link / Juniper / Aruba / HP / Arista / Huawei / Dell / MikroTik / Ubiquiti / NETGEAR / SonicWall / Synology / QNAP / APC / Eaton / TRENDnet / HikVision. First match wins.
  2. Brand keyword + fuzzy — if no model regex hit, scan 26 vendor keyword lists with token-level Levenshtein distance. Edit budget scales with keyword length so Csco → Cisco, Mikrorik → MikroTik, Unfi → UniFi.

Output

{
  "position": "U10",
  "make":     "Mikrotik",
  "model":    "CRS326-24G-2S+RM",
  "version":  "10.5.0.7",
  "raw_text": "Mikrotik Cloud Router Switch CRS326-24G-2S+RM",
  "ocr_conf":   0.74,
  "match_conf": 0.88,
  "source":   "ocr_full"   // ocr_full | ocr_make_only | ocr_failed
}

Honest accuracy on the 12-image test set

MetricPer-device

Vendor recall is OK. Model recall is poor — when OCR returns text, it usually reads the brand and buzzwords (Cloud Router Switch, Mikrotik) but mangles the model number (CRS3Zo instead of CRS326). No regex fix recovers text that wasn't captured. The manual-fallback in Switch Info covers this.

06

Port identification

Team

For each detected switch, count ports and classify them as RJ45 (copper) or SFP (fiber/DAC cage).

  • Counter: port_count.pt YOLO runs on each chassis crop, outputs bboxes per port. Cluster by Y-coordinate to find rows; sort by X within rows; SFP cages have wider pitch + slightly different aspect ratio than RJ45.
  • Pattern refinement: pipeline/port_pattern.py applies vendor-specific naming. TP-Link: 1/0/1..1/0/24. Cisco: Gi1/0/1, Te1/0/1. Required so the SSH probe interface names match.
  • Highlight-a-port flow: POST /api/select with device_index + port draws a circle on the original photo at the exact port location — useful for "go look at port 15 on switch X" tickets.

Port type classification

Three signals, in priority:

  1. Switch's own report: TP-Link's show interface status has an Active-Medium column (copper/fiber). Authoritative.
  2. Cisco-style interface naming: Te/Fo/Hu = SFP+/QSFP/QSFP28, Gi/Fa = copper.
  3. CV port count + SFP count from the chassis bbox: a 24+4 switch has the last 4 entries as SFP.

Key files

  • pipeline/port.py — port-count YOLO + clustering
  • pipeline/port_pattern.py — vendor-aware naming
  • Models/port_count.pt — YOLO weights
  • client/src/pages/PortsPage.jsx:18, 73 — JS mirrors of classifyPorts + shortLabel

08

Firmware — latest version, CVEs

Shared

Three questions answered per switch: what's the latest published version, are you up to date, any known CVEs in your current version.

How latest is detected

  1. Use the vendor domain alias map to identify which domains to search (Aruba's modern docs are on arubanetworking.hpe.com, not arubanetworks.com).
  2. DDG search site:domain MODEL release-notes across multiple alias domains in parallel.
  3. For each candidate URL: fetch, parse, walk a 5-tier extraction (title → h1/h2/h3/h4 → "latest"-keyword proximity → labelled body text → anywhere). First tier with valid versions wins; highest tuple within tier is "latest".
  4. Sanity check: reject the latest if its major version is wildly incompatible with current (catches docs-system version noise).

The IP-vs-version trap

An earlier bug had an Aruba switch claiming "Latest: 10.100.222.115". That's an example IP from a config snippet on the page — not a version. The fix: 4-octet all-≤255 strings are rejected unless found in a labelled position (title, heading, near "version:" keyword).

Major-version sanity check

Even after the IP fix, the scraper sometimes picks up an unrelated docs-system version like 2.5.8 on a Confluence page. Comparing (10,8,1000) >= (2,5,8) would technically be true, so the UI would say "Up to date" incorrectly. Fix: reject latest when its major is wildly mismatched. Threshold: latest major must be at least half of current major, no more than 5x.

CVEs from NIST NVD

Free NVD API. Four-tier keyword fallback (vendor+model+version → vendor+model → model alone → vendor+"switch") because over-specifying returns nothing for niche gear and under-specifying is too noisy. Each CVE flagged with matchesCurrentVersion when description contains the version verbatim.

Demo line "We do best-effort latest-version detection from public release notes — when we can't tell confidently, we say 'couldn't read latest, check vendor' rather than guess. The CVE list is from NIST NVD; we flag CVEs whose descriptions name your exact version."

Key files

Specifications — vendor scraping

Team

Given vendor + model, find the vendor's product page and read the spec table off it. No hardcoded specs database — every call is live; staleness is the vendor's problem.

URL discovery

  • Search across the vendor's known domains (alias map for Aruba, Dell, etc.) plus the Excel-listed URL.
  • Score every result: products page +25, datasheet +25, collateral +30, switches/series +10-15. Penalize forums (-100), EOL pages (-80), shop categories (-30), PDFs (-50).
  • Try each candidate in score order. First page with parseable specs wins.

Spec extraction

Generic — walks every <table> and <dl> on the page. A "spec table" has ≥3 rows of key/value pairs, keys are short, no SKU-shaped strings (those are accessory tables, dropped via _is_sku_list_table).

Open-web fallback

When vendor's own pages have no parseable specs (modern Cisco /site/ pages are JS-rendered), search the open web for distributor / aggregator pages (router-switch.com, FS.com) and try those.

What can go wrong "Product page not found" = search returned no candidates. "No specifications found on any candidate URL" = candidates returned but extractor couldn't parse them. The biggest failure mode is JS-rendered vendor pages; headless-browser fallback (Puppeteer) would fix it but adds latency.

Key files

  • pipeline/all_vendor.py — entire scraper
  • Switch_Vendors_Websites.xlsx — vendor list
  • server/app.js:3304 — /api/specs

Topology — 3D rack view

Team

3D model of the rack with detected devices and the cables between them. Orbit, zoom, switch between 2D (tier graph) and 3D, filter cables by type, capacity heatmap, trace mode (tap two devices, see the cable path).

Why 3D matters

"How is this rack actually wired?" is hard to answer from a flat list. A senior tech glancing at the 3D for 5 seconds spots that the patch panel on U37 has 24 cables to the switch on U36 and 6 to a server on U33 — that's a wiring story.

Data sources

  • CV scan (devices, positions, port counts)
  • SSH probe (which ports are connected on the live switch)
  • Patch-panel registry (manually configured panel-to-panel cabling)
  • LLDP from netdisco (when the switch advertises its peers)

Modes

  • 3D view — Three.js + React Three Fiber, lazy-loaded. Devices as 1U slabs by U-number; cables as tubes routed through patch panels.
  • 2D view — SVG tier graph: core / distribution / access / endpoint. Aggregated edges with cable counts.
  • Capacity heatmap — devices coloured by free-port percentage (green >50%, amber >25%, red <25%).
  • Trace mode — BFS over the device-pair adjacency graph, highlights the shortest cable path.
Multi-rack is data-only Topology JSON has crossRackEdges and neighbors fields, but a "room view" rendering N racks with cross-rack cables doesn't exist yet. Pending

Key files

  • client/src/pages/TopologyScene3D.jsx — 3D scene

Available ports — SSH probe

Shared

Open an SSH session to the switch, run show interface status, parse the output, classify each port as connected / free / reserved.

Verdict logic

  • linkup/connected/up → used
  • err-disabled/shutdown/admin down → reserved (administratively held out of service)
  • Otherwise: a description means "someone planned this port" → reserved; no description → free

The summary card (visually appealing version)

Hero badge with the available count, ETH and SFP chips, gradient utilization bar, expandable list of every available port. The card was iterated from a plain-text "available ports" line into the visual hierarchy that's there now.

Probe lifecycle

  • Fired the moment the user taps "Analyze Rack" — fire-and-forget, idempotent. By the time they navigate to Ports tab, result is usually ready.
  • Cached to localStorage on success. Re-visiting the tab renders from cache; the previous behaviour of re-probing on every nav was removed.
  • Force-refresh via the Retry button on error.

Key files

  • client/src/utils/portsProbe.js — state machine
  • client/src/pages/PortsPage.jsx — UI
  • server/app.js — runSwitchCommand, /api/switch/console/run

SFP advisor

Team

For switches with SFP cages, recommend transceiver part numbers. Slot type (SFP / SFP+ / SFP28 / QSFP+ / QSFP28 / QSFP-DD), primary recommendation, budget option, alternatives, cable types, buy/view links.

Slot-type detection (3 stages, in priority)

  1. Scrape the vendor datasheet for "SFP" / "QSFP" / "Form factor" / "Cage type" rows.
  2. Infer from model name: 400G/QSFP-DD, 100G/QSFP28, 40G/QSFP+, 25G/SFP28, 10G/SFP+, default SFP.
  3. Infer from interface name: Hu → QSFP28, Fo → QSFP+, Te → SFP+, Gi → SFP.

No hardcoded part-number database

Every recommendation is live: open-web search for compatible transceivers, scrape part numbers + prices, score by brand authority + price availability + spec completeness. Fallback when offline: cable-type guidance only (engineering fact, not scraped data) — no specific SKUs.

Key files

  • pipeline/sfp_recommend.py — server scrape + rank
  • client/src/utils/sfpDatabase.js — client cache + offline fallback
  • client/src/pages/PortsPage.jsx:SfpAdvisor — UI

Netdisco proxy

Team

Netdisco is an open-source SNMP / LLDP discovery tool — point it at a network range, give it SNMP read-only credentials, it walks every switch and router and learns who's connected to whom.

We run a netdisco container in netdisco-docker/, and the app proxies a small set of read-only queries through to it. The data shows up in the Network tab on the Results page.

Routes

GET /api/netdisco/healthcontainer health
GET /api/netdisco/devicesall known devices
GET /api/netdisco/device/:ipone device's record
GET /api/netdisco/device/:ip/portsper-port info
GET /api/netdisco/device/:ip/neighboursLLDP/CDP peers
GET /api/netdisco/rack/:rackId/neighboursunion per rack
POST /api/netdisco/sync/:rackIdtrigger a re-walk

Source-merge with scan / CMDB

When all three sources (scan, netdisco, CMDB) claim "switch X is at U10", the reconciler matches by serial → MAC → IP+position → position alone. Priority on conflict: CMDB → scan/OCR → netdisco. Netdisco's vendor strings (parsed from SNMP sysDescr) are the least-trusted because they vary across firmware revisions.

Retraining — closing the loop

Me

Final stage of active learning. Poll per-model queues, kick off training when threshold crossed, validate on holdout, promote only if better.

Lifecycle of one retrain

  1. Check thresholds. For each model, is the queue size over its retrain threshold?
  2. Export the queue. Dump every queued sample (image + label) into runs/<model>-<timestamp>/.
  3. Train. Spawn the matching trainer subprocess (YOLO for devices, EfficientNet for cables). Reads dataset.jsonl, produces best.pt + val_metrics.json.
  4. Validate. Run new model against the frozen holdout/<model>/ set. Get back the primary metric (mAP@0.5 for devices, macro-F1 for cable, RMSE for port_count).
  5. Promotion gate. Compare new vs production. Promote only if better by at least the configured margin.
  6. Promote or shelve. Promoted: copy best.pt to Models/, update registry, mark queue exported, touch promoted.flag. Shelved: leave run dir for inspection, queue stays pending.

Non-destructive

Production model isn't touched until the new one passes the gate. A bad batch of corrections produces a bad candidate that fails validation and never reaches users.

Multi-tenancy

Me

Single deployment serves multiple customers safely. Customer A's racks invisible to Customer B and vice versa. Mechanism: every user belongs to one tenant; every :rackId route checks ownership.

Schema

CREATE TABLE rack_owners (
  tenant_id  INTEGER NOT NULL,
  rack_id    TEXT    NOT NULL,
  created_by INTEGER,
  created_at TEXT,
  PRIMARY KEY (tenant_id, rack_id)
);

Public API

const tenant = require('./lib/tenant');
tenant.claimRack(tenantId, rackId, userId);     // INSERT OR IGNORE
tenant.tenantOwnsRack(tenantId, rackId);        // → bool
tenant.tenantRackIds(tenantId);                 // → Set<string>
tenant.listRacksForTenant(tenantId, limit);     // → list
tenant.requireRackOwnership(req, res, next);    // Express middleware

The shared-image case

Two tenants who scan the same physical rack get the same rackId (SHA-256 of image). The outputs/RK-XXX/ folder is shared on disk (efficient, we don't re-run the pipeline). But the API surface treats each tenant as if they have their own rack — neither tenant can see the other has also scanned it.

404 instead of 403 on ownership miss

403 leaks information ("this rack exists, you can't see it"). 404 is indistinguishable from "never scanned by anyone" — privacy by API design.

Auth + Secrets

Shared

JWT user authentication plus AES-256-GCM encrypted SSH credentials.

User auth (JWT)

const token = jwt.sign(
  { sub: user.id, tenant_id: user.tenant_id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: '7d', issuer: 'racktrack' }
);

Three roles: viewer < engineer < admin. requireRole(name) middleware enforces. Composes with tenant.requireRackOwnership.

SSH credential encryption

Two halves in two different places:

  • Encrypted blob in server/.env as SSH_CREDS_ENC=<base64>. Safe to check into a private repo — useless without the key.
  • Key in either SSH_CREDS_KEY env var (preferred) or server/.env.key (32-byte hex).

Layout of the encrypted blob: [12-byte IV][16-byte auth tag][N-byte ciphertext]. Plaintext is JSON with defaultUser, defaultPassword, per-host overrides, last successful host.

ServiceNow auth is HTTP Basic self.auth = (os.environ["SN_USER"], os.environ["SN_PASSWORD"]) — affects 8+ scripts in servicenow/. Move to OAuth client-credentials grant. Pending

Architecture — how everything fits

Team

Three independent processes. Each has a clear job.

  • Client — React 19 + Vite 6 SPA, Capacitor 7 wrapper. Android APK today; iOS soon. Three.js for 3D. Calls the server over HTTPS with a JWT.
  • Server — Express 4 on Node 20. ~3,900 LOC of server/app.js + 6 routes from cmdb_ticket_proxy.js + 7 from netdisco_proxy.js = 58 total HTTP routes. SQLite for users/tenants. Filesystem state under outputs/.
  • Pipeline — Python modules under pipeline/, servicenow/, active_learning_Cache/, retraining_learning/. Each invocation is a subprocess that reads inputs and prints one JSON line of output.

Why the split

The heavy work (CV, OCR, scraping) is Python-shaped because the libraries are Python. HTTP/auth/session work is Node-shaped because Express is good at that. The two talk through subprocess stdout/stderr.

Known limits — what doesn't work

Shared (the honest scorecard)

Hard accuracy numbers

Metricv1 baselineCurrent
Devices CV-classified Switch/Router/Server/Firewall40108
Of those, vendor identified17 (42.5%)22 (20.4%)
Of those, model identified1 (2.5%)2 (1.9%)

Per-photo, 8 of 12 produce a vendor; 0 produce a fully-correct model.

What you can honestly say in a demo:

  • "We auto-detect about half the vendors on a clean photo, and we have a manual fallback for the rest."
  • "Model number recall is poor today — most fascia text isn't legible enough at typical phone-camera resolution. We're closing that gap with active learning."

What you should not say:

  • "It identifies switches with 95% accuracy."

Bugs we know about and have not fixed

  • feedback.jsonl grows without bound — no rotation. Eventually gigabytes.
  • outputs/<rackId>/ grows without bound — same. Add a GC job that prunes folders with zero rack_owners rows.
  • Cisco / modern Aruba pages are JS-rendered — static-HTML scraper finds nothing. Headless-browser fallback would fix it.
  • CVE counts aren't version-applicable — NVD's free API takes free text. matchesCurrentVersion is a hint, not a precise filter.
  • Recall-gap banner removed — side_labels.py still works server-side, the UI banner had broken matching logic and was removed.
  • Manual entries don't flow to ServiceNow — synth.py doesn't read localStorage overrides. Tracked.
  • Multi-rack topology is data-only — JSON has fields, no room view rendering yet.
  • Dev-approve bypass — authenticated, but auditors flag it. Production fix: gate behind NODE_ENV.
  • ServiceNow auth is HTTP Basic — 8+ scripts. Move to OAuth.
  • Worker pool doesn't reload models on promotion — restart Node after a promotion.

Roadmap by category

Detection accuracy: PaddleOCR fallback (recover the 4-of-12 zero-OCR cases), tighter photo-quality gate, active-learning operating cycle automation.

Multi-rack: Room view rendering, floor-plan import.

Mobile / native: iOS build, AR overlay native impl, on-device inference.

Platform: Per-tenant rate limiting, GDPR delete-my-data flow, Postgres migration.

Operations: Hot-reload model weights, cron-driven retrain, OTel exporter.

What we explicitly chose NOT to build

  • Real-time CMDB sync (writes flowing back from ServiceNow). Cost > benefit at current customer count; explicit re-scan is acceptable.
  • In-house CVE database. NVD is free and we don't add value mirroring it.
  • Vendor-specific spec extractors. Generic table extraction handles most; the long tail (modern JS-rendered) gets headless-browser fallback, not bespoke per-vendor code.
  • iOS-first before Android stability. Most rugged datacenter phones are Android.
