RackTrack · Plain-English Brief

What's working right now

A no-jargon, spoon-fed walkthrough of where we ended up after two days of work. What we tried, what failed, what's running on your phone right now, and what's left for tomorrow.

14 May 2026 · For your own notes

1The goal — in one sentence

Make the RackTrack app's rack identification work without a server, without internet, on the phone itself. Take a photo, find the switches, find a specific port — all locally.

Why this matters: a technician in a basement, on a customer's secured network, or anywhere with bad Wi-Fi can still scan a rack and look up a port. No waiting for the server, no upload, no privacy concerns about sending photos.

2Where we started — yesterday

Yesterday we proved the AI models could run on a phone at all. We used something called WASM (a way to run code inside the app's web view, like a browser plugin). It works — but it's slow. One scan of a rack took about 149 seconds. Too slow to ship.

We also shrunk the models from 467 MB down to 206 MB using a technique called dynamic INT8 quantization — basically swapping 32-bit numbers for 8-bit numbers in the model file. Same idea as compressing an image from "full quality" to "good quality" to save space.

3What we tried today — eight dead-ends

Today the goal was to make it fast. The plan: use native code instead of WASM. Native means talking directly to the phone's processor, which is 10-20× faster.

Sounds easy. Wasn't. Here's every attempt in order:

1

Try yesterday's INT8 models with native code

Result: the app froze for 3+ minutes trying to load even one model. The native code couldn't handle these specific models.

2

Re-quantize models with "static INT8" instead of "dynamic INT8"

Result: models loaded fast (137 ms!) but produced literally all zeros as output. The detector found nothing. Bad quantization.

3

Try different INT8 settings (per-channel scales, entropy calibration)

Result: still all zeros. The whole INT8 path was broken for these YOLO models.

4

Add a recommended pre-processing step before quantizing

Result: still all zeros. Ruled out "missing step" as the cause.

5

Switch to a different INT8 file format ("QDQ")

Result: produced an invalid file that wouldn't even open. Dead end.

6

Try FP16 (half-precision floats) via a converter library

Result: the converter library broke specific parts of YOLO models. Wouldn't load.

7

Try FP16 via Ultralytics' built-in tool

Result: silently produced FP32 files instead of FP16 (it needs a GPU we don't have to actually do FP16). Looked like it worked but didn't.

8

Just ship the original FP32 models, no shrinking

Result: this worked. Bigger app, but everything runs correctly and fast.

What we learned: the INT8 path failed because YOLO's "detection head" (the last few layers that produce confidences between 0 and 1) is too sensitive to be quantized. We discovered this late in the day and proved it works if we tell the quantizer to skip that specific part. That's tomorrow's fix.

3What's working right now

Here's what you can do on the phone today, with no internet, with the APK we built:

✓
Open the app offline The login page has a green "Continue offline" pill at the bottom that takes you straight to On-Device Scan even if the server is down. Auth no longer auto-logs-out on network errors.
✓
Three ways to feed an image Take a photo with the camera, upload one from your gallery, or use the bundled sample rack to test without taking a photo.
✓
Detection runs locally in ~5 seconds The app uses two AI models — one finds devices on the rack, the other finds ports on each device. Both run on the phone's CPU using a native AI runtime. No server call, no network.
✓
Only switches and patch panels show up Servers, firewalls, PDUs, etc. get filtered out of the device list. Those don't have ports to find, so they'd just clutter the dropdown.
✓
Pick a device from a dropdown After the scan, a dropdown lists every detected switch and patch panel: "Device 1 · Switch · 24 ports", sorted top-to-bottom like they appear on the rack. Pick one and it gets a thick blue outline on the rack image.
✓
Type a port number, see it highlighted Type "14" (or whatever port you need), tap Find, and that exact port gets a yellow outline and a "Port 14" label drawn directly on the rack image. Status line confirms: "Port 14 located on Device 2 (24 ports total)."
✓
Detection matches the server Same model files. Same thresholds (0.20 confidence for devices, 0.23 for ports, 0.45 NMS — straight from config.json). If the server can see a device, the phone can too.

4What's still TO DO

★
Tomorrow: try FP16 to shrink the app The APK is 860 MB because we shipped uncompressed FP32 models. FP16 ("half-precision") is the same math but stored in half the space, with essentially zero accuracy loss. Should drop the APK to about 330 MB at zero risk to detection quality.
★
Static INT8, properly fixed Today we found the cause of the INT8 zero-output bug — YOLO's detection head can't be quantized, but if we skip that part it works. We proved it in Python. APK would drop to ~260 MB at ~1% accuracy drift. Worth a try after FP16 to compare.
★
NNAPI / GPU acceleration The phone has hardware specifically for AI math (Neural Networks API, GPU) we're not using yet. Hooking it up would make inference another 3–5× faster — full scan under 1 second. Needs caching of the first-time compile step.
★
Live camera mode Right now you take a still photo, then it processes. With NNAPI in place, the app could run detection on every camera frame and draw the boxes live as you move the phone. Like augmented reality for racks.
★
Sync to server when network's back The phone does detection but can't do OCR (read device labels like "SW01-CORE") or CMDB lookup. Plan: store offline scans locally, upload them when the phone comes online so the server can fill in the labels and reconcile with ServiceNow.
★
ML Kit OCR for device labels Alternative to the above: run OCR directly on the phone using Google ML Kit. That removes the "Device 1, Device 2" placeholders and shows real switch names. No server needed.

6The bits worth showing in a demo

  1. Turn off Wi-Fi and mobile data. Show the phone is fully offline.
  2. Open the app, tap the green "On-Device Scan · works offline" pill on Home.
  3. Tap "Try with sample rack" (or take a real photo). Show the scan running.
  4. When it finishes, point at the rack image — every switch and patch panel has a green box. The dropdown lists each one.
  5. Pick a device. Show it gets a thick blue outline.
  6. Type a port number, tap Find. Show the yellow port box and the "Port N" label appearing on the rack image, on the right device.
  7. Notice the phone is still in airplane mode. Drive the point home.
