RackTrack · On-Device Scan

The full journey — from "can the phone do it?" to a working offline scan

Two days of work, eight dead-ends, ten fixes that landed. This page is the complete story in plain English — every approach we tried, why each one failed or worked, and what every technical term means.

RackTrack · On-Device Scan write-up · 14 May 2026

1.What we were trying to do

Today RackTrack works like this: you take a photo of a rack with your phone, the photo is uploaded to a server, the server runs nine AI models on it to figure out what's in the rack, and sends the answer back. The phone is basically a camera + display; all the real work happens on the server.

The question we wanted to answer: can the phone do all that work itself, with no server and no internet? That would mean technicians can scan racks in basements, datacenters with no Wi-Fi, customer sites without VPN — anywhere.

Plain English

What's "the pipeline"?

We say "the pipeline" or "the scan flow" a lot. It's the chain of AI models that look at one rack photo and produce the structured answer: where the rack units are, where the devices are (switches, patch panels, servers…), where each port is, what type each port is. Nine separate models, each doing one job, run one after another.

Plain English

What's a "model"?

A model is a file (usually ending in .pt or .onnx) that contains a trained neural network — millions of numbers that were tuned by training on thousands of rack photos. When you give the model a new photo, it does math on those numbers to produce boxes around the things it was trained to find. Our biggest model is ~175 MB, our smallest is ~16 MB.

2.Yesterday — proving the phone can run the models at all

Before trying to make it fast, we just wanted to confirm the models could run on a phone at all. We chose the easiest possible runtime to start with: WASM, which lets you run AI models inside a regular web page — the same web page that's already inside our Capacitor app's WebView.

Technical term

WASM (WebAssembly)

A way to run compiled code (like C++) inside a web page. Slower than native code but works anywhere a browser works. ONNX Runtime ships a WASM build, so you can load .onnx models and run them from JavaScript with no platform-specific code.

Technical term

ONNX & ONNX Runtime (ORT)

ONNX is a portable file format for neural networks. ONNX Runtime (ORT) is the engine that takes one of those files and actually runs it. Models are originally trained in PyTorch (.pt files); we export them to .onnx so they can run on phones without needing PyTorch installed.

Step one — shrink the models

The original nine PyTorch files together were 467 MB. Too big to bundle into a mobile app. We used a technique called dynamic INT8 quantization to shrink them to 206 MB — a 56% reduction.

Technical term

Quantization

Neural networks store their weights as 32-bit floating-point numbers (FP32). Quantization reinterprets those weights as 8-bit integers (INT8), making the file 4× smaller. The math becomes slightly less precise, but for most models the accuracy loss is tiny.

Dynamic INT8 means the activations (the values that flow between layers as the model runs) get quantized on-the-fly at inference time. Static INT8 pre-computes calibration data using sample images and bakes the quantization scales into the file — smaller and faster, but harder to get right.

Step two — see if it actually works on a phone

We bundled the nine INT8 models into the app, added a new "BenchmarkPage" that runs the full pipeline on a photo, and tested it on a real Samsung Galaxy A35 5G.

Result: it worked. Every model loaded and produced correct output. But it was slow — 6.4 seconds for a single model, 149 seconds total for the full nine-model pipeline. Too slow for a real scanning workflow.

Yesterday's status

Yes, the phone can run the pipeline. But WASM is the slowest possible runtime — like running a sports car in first gear. To make it fast we need native ORT, which talks directly to the phone's CPU instead of through a browser-style runtime.

3.Today — the rabbit hole of dead-ends

Native ORT promised to be 10–20× faster than WASM. The plan was simple: write a small Capacitor plugin that loads our existing .onnx files and runs them through onnxruntime-android instead of the WASM build. Each model would then run in a few hundred milliseconds instead of seconds.

Reality: eight different things failed before one worked. This is the table-of-shame, in the order we hit them. Each was a real discovery — none of them is obvious from the docs.

# What we tried What broke
1 Dynamic INT8 models (yesterday's, that worked in WASM) + native ORT Loading the model hung the app for 3+ minutes. Trying different "optimization levels" (ALL_OPT, BASIC_OPT, NO_OPT) didn't help.
2 Static INT8 with QOperator format + per-tensor scales + MinMax calibration Loaded in 137 ms (great!) but produced a tensor of literally all zeros. The detector found nothing.
3 Static INT8 with QUInt8/QInt8 + per-channel + Entropy calibration (the "good recipe") Still all zeros. Quantization tooling was fundamentally choking on these YOLO models.
4 Added quant_pre_process (the standard ORT preprocessing step) before quantizing Still all zeros. Eliminated "missing step" as the cause.
5 Switched to QDQ format (different way of representing INT8 ops) Produced a syntactically invalid ONNX file. Wouldn't even load.
6 FP16 via onnxconverter_common library Broke the Resize nodes (they're used in YOLO's feature pyramid). Wouldn't load.
7 FP16 via Ultralytics' built-in half=True export flag Silently produced FP32 files (no real conversion). Ultralytics needs a CUDA GPU for actual FP16 export.
8 Just ship the raw FP32 ONNX files ✅ It worked.
Technical term

Optimization level

Before running a model, ORT can rewrite the model's internal graph to be faster — combining adjacent operations, removing redundant ones, etc. There are three levels: NO_OPT (no rewrite), BASIC_OPT (safe rewrites), ALL_OPT (aggressive rewrites including BatchNorm folding — the biggest speed win for our models, 9× faster runtime).

Technical term

QOperator vs QDQ formats

Two ways INT8 models can be stored. QOperator uses special "QLinearConv" / "QLinearMatMul" ops that do everything in INT8 directly (fast, but requires the runtime to support those ops). QDQ ("Quantize-Dequantize") keeps the model mostly in FP32 but adds Q/DQ nodes that say "convert to INT8 here, then back to FP32 after this op" (more portable, but slower and adds many extra nodes).

Technical term

Calibration (in static quantization)

To go from FP32 to static INT8, you have to figure out a "scale factor" for each tensor: how do we squeeze a wide range of FP32 numbers into INT8's narrow range without losing too much precision? You answer this by running the FP32 model on a small calibration set of representative images and watching the actual numbers that flow through. There are different methods:

  • MinMax: just track the smallest and biggest values seen.
  • Entropy (Kullback-Leibler): build a histogram and find scales that minimize information loss.
  • Percentile: ignore the top/bottom N% of values to avoid outliers.

Entropy is usually the best balance of accuracy and effort, but for YOLO detection heads on our specific models, none of these worked.

Why the INT8 attempts all produced zeros

Most likely cause: YOLO's detection head contains a sigmoid activation that maps logits (raw scores) into confidences between 0 and 1. When the input scale is wrong, sigmoid saturates — every value gets clamped to almost-exactly-zero or almost-exactly-one. Combined with the QOperator kernels not having exactly the right ops for our model's specific operator chain, the output collapsed to all zeros.

Fixing that properly requires either retraining the model with quantization-aware training, or building a custom calibration pipeline that targets the detection head specifically. Neither was tractable in a one-day spike.

4.The pivot — just ship FP32

After eight INT8/FP16 dead-ends, we tried the simplest possible thing: use the original FP32 ONNX files (no quantization at all) and run them natively. The trade-off is size: total bundle grows from 206 MB to 774 MB. The win: it actually works correctly and is still massively faster than WASM.

Trade-off

774 MB of bundled models is a lot. Most production phones can install it, but it's not great. Future work: fix static INT8 properly (which would land us back at ~200 MB and give another 2-3× speedup on top of FP32). For now, FP32 works and ships.

5.The ten fixes that made it actually work

Even with FP32, just dropping the files in didn't work. Each of these was a separate discovery before the pipeline ran end-to-end with correct results. If any one is missing, the whole thing falls over.

1. FP32 ONNX files, kept under their old _int8.onnx names. The JavaScript side already knew the filenames — kept them as-is so we didn't have to rename every reference. The content is now FP32, despite the filename.
2. OptLevel.ALL_OPT for graph optimization. The biggest single speed win: 9× faster inference. Without this, Pass 1 inference was 6+ seconds per model; with it, ~1 second.
3. AssetFileDescriptor.getLength() for byte-array preallocation. Reading the 174 MB Device_final.onnx into a growing ByteArrayOutputStream doubled capacity 15 times and OOM-ed Android's heap. Preallocate exactly the right size and the load finishes instantly.
4. EXIF rotation via android.media.ExifInterface. Phone cameras store landscape pixels with a "please rotate me 90°" EXIF tag. The WebView respects it; BitmapFactory does not. Without applying EXIF in the plugin, the model saw the rack sideways and the box overlay looked like vertical stripes.
5. Letterbox preprocessing. YOLO models are trained on square-padded input, not stretched. Plain resize squashes a portrait rack and the device detector misses ~90% of devices. Letterboxing adds gray padding to make the input square before resizing.
6. noCompress 'onnx' in Gradle's AAPT options. Without this, the Android build tries to compress 200+ MB of binary ONNX files at packaging time and OOMs. ONNX is already binary so compression doesn't help anyway.
7. Gradle JVM heap bumped to 6 GB. Adjacent to #6 — the build itself needs enough memory to hold the asset table during APK packaging.
8. Multi-class YOLO decode. This was a hidden bug in our JavaScript decoder. YOLOv8 outputs [1, 4+C, N] where C is the number of classes. Confidence per box is max(class_scores), not just class_scores[0]. We were reading only channel 4, which dropped every detection of class 1..C-1 silently. That's why best_33 (18 classes) returned 0 detections and the device detectors returned 1 each.
9. Confidence and NMS thresholds matching the server. devices_conf=0.20, ports_conf=0.23, NMS_IoU=0.45 — same as config.json on the server.
10. CLI build-and-install loop. JAVA_HOME=<Android Studio's JBR> ./gradlew assembleDebug + adb install -r, driven by UI Automator dumps + adb shell input tap. Means we can iterate without manual phone taps and run the same test ten times in ten minutes.
Technical term

Letterboxing

Before you feed a photo to a YOLO model, the photo has to be resized to exactly 640×640 pixels. A naive resize just stretches the image to fit, which distorts everything. Letterboxing keeps the aspect ratio: it scales the image so the longest side fits, then pads the shorter side with gray (114,114,114) to fill the square. The same way old movies are shown on a TV without cropping or stretching.

Technical term

Confidence threshold

Each box the model predicts comes with a confidence score between 0 and 1. The threshold is the minimum confidence we'll accept — anything below gets thrown away. Higher = fewer false positives but more missed real detections. Lower = catches more real detections but lets more junk through. 0.20–0.25 is typical for production deployments.

Technical term

NMS (Non-Maximum Suppression) and IoU

YOLO models predict thousands of overlapping boxes per image; you usually want one box per real object. NMS removes duplicates: take the highest-confidence box, throw away any other box that overlaps it heavily, repeat.

"Heavily" is measured by IoU ("Intersection over Union") — the overlap area divided by the combined area. Two identical boxes have IoU = 1.0; two non-overlapping boxes have IoU = 0.0. NMS at IoU=0.45 means "discard any box that overlaps a kept box by 45% or more."

Technical term

EXIF rotation

When you take a portrait photo with a phone, the camera's image sensor is actually mounted in landscape. To save flipping pixels at capture time, the JPEG file stores landscape pixels plus an EXIF orientation tag that says "viewer: please rotate me 90° clockwise." Most image viewers respect this; Android's BitmapFactory doesn't, by default. So we had to read the tag ourselves and rotate the bitmap before running the model on it.

6.What works now

The flow on the phone

  1. Tap "On-Device Scan · works offline" on the Home page
  2. Pick an input: camera, the bundled sample rack, or upload an image from gallery
  3. The page runs the device detector (best_32) on the whole image
  4. For each detected Switch or Patch Panel, it runs the port detector (port_best) on that device crop
  5. The rack image appears with green boxes around every detected device
  6. A dropdown below the image lists each device: "Device 1 · Switch · 24 ports"
  7. Pick a device → the selected device gets a thick blue outline on the rack image
  8. Type a port number → the matching port gets a yellow outline + "Port N" label drawn on the rack image

The simplified production flow

The benchmark version of this page runs all 9 models and shows timing for each. The user-facing version we shipped today runs only 2:

  • best_32 — detects devices on the whole rack (the server's primary device detector)
  • port_best — detects ports on each detected device crop (the class-aware port classifier)

Skipping the other seven (alternative unit/device detectors, port-type classifier, port-count locator) is what gets the wall-clock down from ~27 s to ~5 s for a typical rack. The full nine-model version is preserved in branch on-device-inference-writeup as the diagnostic / evaluation tool.

7.Server parity

The on-device pipeline now uses the same model files and the same thresholds as the server, so the answers it produces should match /api/analyze for the visible rack content. Specifically:

  • Same models. best_32.pt and port_best.pt exported to ONNX FP32 — bit-for-bit the same weights.
  • Same letterbox preprocessing. Aspect-preserving resize + gray (114,114,114) padding, just like Ultralytics' Python library does.
  • Same confidence threshold (devices_conf=0.20, ports_conf=0.23, from config.json).
  • Same NMS IoU (0.45, the Ultralytics YOLOv8 default).
  • Same class filtering applied to the device detector output (only Switches and Patch Panels enter the port-finding flow).
What this means

If the server can see a device, the phone can too — and at the same coordinates. Minor differences may remain (FP32 math on x86 vs ARM, bilinear interpolation library differences) but they're at the sub-pixel level, not at the device-count level.

8.What's not on the phone (and why)

The on-device scan is the detection part of the server pipeline. The other things the server does are genuinely network-dependent and stay on the server:

  • OCR for device labels. The server runs EasyOCR over each device crop to read printed labels (e.g., "SW01-CORE"). Running EasyOCR on a phone is possible (via ML Kit) but a separate workstream.
  • Vendor matrix / firmware lookup. Maps detected model numbers to vendor + recommended firmware. Lives in a server-side spreadsheet.
  • CMDB reconciliation. Compares the scan to what ServiceNow's CMDB thinks is in this rack, flags drift, generates work-note tickets. This requires ServiceNow API access, which is server-only.
  • SSH switch probe. Logs into the actual switches over the network to read live port state. The phone is rarely on the same LAN as the switches.

The intended flow: technician scans offline, gets device/port locations instantly on-device, then when they're back on the network the offline scan syncs to the server which fills in the labels, firmware, and CMDB reconciliation parts.

10.What's next, in order of payoff

  1. Static INT8, fixed. Get past the all-zeros bug and we're back to ~200 MB bundle (instead of 774 MB) and 2–3× faster. Tractable; needs ~1 day of focused work on the quantization tooling.
  2. NNAPI / GPU delegate. The phone has hardware AI acceleration we're not using. ORT supports it via the NNAPI execution provider. First-run compile cost is real (~30 s/model, cacheable to disk), but steady-state inference becomes another 3–5× faster. Could put us under 1 second wall-clock.
  3. Continuous / live-camera scan mode. Re-run the pipeline every few frames from the camera preview, smoothly update the overlay. Needs steady-state inference under ~500 ms, which depends on #2.
  4. Re-sync to the server when online. Cache offline scans locally, push them to /api/analyze when the network is back so OCR, CMDB reconciliation, and firmware lookup can fill in. Pure plumbing — no ML.
  5. ML Kit OCR for device labels. Reads switch labels ("SW01-CORE") on-device, no server. Replaces "Device 1, Device 2" with real names.
