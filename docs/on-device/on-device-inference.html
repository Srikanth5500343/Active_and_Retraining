RackTrack · Decision Brief

On-Device Inference

Verified facts about our current pipeline, what we'd need to validate before committing, and a proposed first experiment.

12 May 2026

1.What we have today (verified)

The current inference pipeline is Python, running server-side. Every scan from the phone goes to /api/analyze, which invokes pipeline/runner.py. The pipeline is multi-stage:

  • best 33.pt — YOLO, detects servers only
  • best 32.pt — YOLO, detects every other device class
  • unit.pt / Units.pt — YOLO, detects rack units (U)
  • port_count.pt — YOLO, locates and counts ports
  • switch_patch.pt — patch-panel port classifier
  • Device_final.pt — YOLO, final device pass
  • best_model_efficientnet.pth — EfficientNet, cable / port-type classifier

2.The proposal

Run the AI on the phone instead of on the server. The models would be converted to a phone-native format (TFLite for Android, CoreML for iOS) and shipped with the app — or downloaded after install.

3.What we know is true

  • The models are convertible in principle. All eight are standard PyTorch — YOLO (Ultralytics) and EfficientNet — both with well-documented export paths to ONNX → TFLite / CoreML.
  • Shipping 396 MB of models in the app install is impractical. Google Play and the App Store warn users above 200 MB on cellular. A post-install model download would have to be built.
  • The pipeline glue is not trivially portable. pipeline/runner.py and its imports (detection.py, port.py, cable.py) contain image preprocessing, multi-pass detection logic, IoU dedup, and unit-assignment code in Python. None of that runs on the phone today — it would have to be rewritten in Kotlin (Android) and Swift (iOS), or in JavaScript inside the WebView.
  • Our test devices have known limits. The Samsung A14 5G is a budget device; the REDMAGIC 10 Pro is a flagship. Neither is ARCore-certified, but both can run normal ML inference. The two together let us measure the gap between low-end and high-end performance.

4.What we don't know yet

These all require measurement, not estimation:

  • Current /api/analyze end-to-end latency in production (median, p95).
  • How long each model takes to run on each test phone after conversion.
  • How much accuracy we lose when we quantize each model (INT8 / FP16).
  • Whether any model fails to convert cleanly (custom ops, dynamic shapes).
  • Our user device distribution — what hardware our actual users run.

Until those are measured, any timeline or "% of users helped" number is a guess. The brief below is structured around finding answers, not predicting them.

5.Proposed first experiment

Goal: get one model running on the phone end-to-end. Measure latency, accuracy, and ease of conversion. Make no further decisions until we have those measurements.

  1. Pick the smallest standalone model. The candidates are best_model_efficientnet.pth (15.6 MB, classifier — runs on cropped patches), switch_patch.pt (21.5 MB), and unit.pt (21.5 MB). Final pick depends on which one we can validate accuracy on quickly.
  2. Convert it. .pt → ONNX → TFLite (Android) and .pt → CoreML (iOS). Record any conversion failures.
  3. Benchmark on our two test phones. A14 5G as the low-end floor, REDMAGIC 10 Pro as the high-end ceiling. Capture median + worst-case inference time.
  4. Compare accuracy. Run the converted model against the original on a held-out set of rack photos. Quantify any drop.
  5. Compare to current server latency. Measure /api/analyze p50 / p95 on the same network conditions our users have. The on-device approach only makes sense if it actually wins on latency.

After step 5, we have real numbers for one model on real devices. Then — and only then — we decide whether to convert the rest.

6.What this brief does not claim

  • It does not claim a target performance ("20 ms", "10× faster"). Those numbers require measurement.
  • It does not claim a percentage of users helped or hurt. That requires user-device telemetry.
  • It does not claim a timeline for the full migration. That depends on the conversion findings.

7.Recommendation

Recommendation

Approve a one-week experiment on a single model. Deliverable: a memo with measured latency, measured accuracy loss, and a converted model artifact. That memo replaces every assumption in this brief with a number.

The full on-device migration is a much larger decision. It should be made on the back of measurements from this experiment — not on the back of estimates.
