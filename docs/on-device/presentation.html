1.What it is

The same rack identification the server has always done — find every device, find every port — running natively on a Samsung Galaxy A35 using two neural network models bundled inside the Android APK. No network round-trip, no API call, no waiting for the server.

The flow on the phone, from the user's perspective:

1

Pick an image

Take a photo with the camera, upload one from gallery, or try the bundled sample rack. All three options are on the On-Device Scan page.

2

The phone runs the device detector

A YOLO model called best_32 scans the whole photo and finds every device. We filter the results to keep only Switches and Patch Panels — those are the ones with ports.

3

The phone runs the port detector on each device

For every detected device, the phone crops that part of the photo and runs a second model called port_best that locates every individual port on that device — RJ-45, SFP, console, etc.

4

You see the result on one image

The rack photo appears with green boxes around every detected device. A dropdown below the image lists each one: "Device 1 · Switch · 24 ports."

5

You ask for a port

Pick a device from the dropdown — it gets a thick blue outline on the rack image. Type a port number, tap Find, and that exact port lights up with a yellow outline and a "Port N" label — right on the rack photo, in place.

What this means

A technician with no internet, in a basement datacenter or a customer site behind their firewall, can identify a specific port on a specific switch in about five seconds with nothing but their phone.

The two models that do the work

Model Job Size Time per call
best_32 Find every device on the rack and classify it (Switch, Patch Panel, Server, PDU…). We keep only Switches and Patch Panels. 175 MB ~2.0 s
port_best Find every port on a single device crop and classify it (main / SFP / console). 103 MB ~1.0 s × N devices

For a typical rack with 4–6 switches/patch panels, that adds up to roughly 2 + 6 × 1 = ~8 seconds total inference, plus about a second of image preprocessing and JS overhead — landing the full scan in ~5–10 s end to end.

What we ship in the app

  • 2 ONNX model files (~278 MB total, bundled inside the APK as compressed assets)
  • 1 native Capacitor plugin (OrtNativePlugin.java, ~270 lines) — the bridge between JS and the AI runtime
  • 1 React page (BenchmarkPage.jsx, the On-Device Scan UI)
  • The onnxruntime-android library (Maven dependency)

Total addition to the APK: roughly 290 MB of models + runtime. The app is fully self-contained — it can be installed on a phone in a Faraday cage and still work.

3.Why this matters

Reliability

Works without internet

Basements, customer sites, datacenters with no Wi-Fi, anywhere a phone has battery — the scan runs locally. No queue at the API server, no flaky cellular.

Speed

No network round-trip

Even on perfect Wi-Fi, uploading a 2 MB photo + waiting for the server queue takes 10–20 s. Local scan finishes in 5.

Privacy

Photos never leave the phone

Important for security-conscious customers, defence/government sites, or anywhere imagery of internal infrastructure must not be transmitted.

Cost

Zero server compute per scan

Inference is free on the phone. Every offline scan is a scan the server didn't have to run, with no GPU cost or worker queue contention.

Determinism

Same models as the server

Same .onnx exports of the same trained PyTorch weights, same letterbox preprocessing, same confidence and NMS thresholds. The phone returns what the server would return.

Resilience

Falls back gracefully

The login page has a "Continue offline" pill that routes straight to On-Device Scan even when the auth server is unreachable. Auth tokens are no longer wiped by network failures.

Phone and server now use identical thresholds and the same model weights, so detection counts match. Any remaining differences are at the sub-pixel level (FP32 math on x86 vs ARM).

Result

14–21× faster than browser-based on-device, and the answers match the server. A full scan-and-find-port flow takes about as long as a phone screen unlock.

5.What it is not (yet)

Some parts of the server pipeline are genuinely network-dependent and stay on the server. The on-device scan delivers detection; the server delivers everything that needs your CMDB / network access.

  • Device labels ("SW01-CORE", "PP-A3") come from server-side OCR. The phone shows "Device 1, Device 2…" until the offline scan syncs to the server.
  • CMDB reconciliation with ServiceNow needs the ServiceNow API and stays server-side. The phone scan can be replayed against the server later.
  • Live port-status probe (SSH into the switch) needs the phone to be on the same LAN as the switch. Out of scope.
  • Firmware lookups against the vendor matrix stay server-side.

Intended flow: scan offline → see results immediately → sync when network's back → CMDB + firmware lookups fill in automatically.

6.What's next

The current build is production-shippable for the offline-scan use case. Three follow-ups will make it materially better:

  1. Static INT8 quantization, fixed. Drops the model bundle from 290 MB to ~120 MB and makes inference 2–3× faster. The all-zeros bug we hit is solvable with one more round of Ultralytics-side export tweaks. Probably one focused day.
  2. NNAPI / GPU delegate. The phone has hardware AI acceleration we're not using. ORT supports it. First-run compile cost cached to disk; steady-state another 3–5× faster. Could put the full scan under 1 second.
  3. Live camera mode. Re-run inference every few frames from the camera preview so the device boxes track the rack live. Depends on #2 to be smooth.
Bundle size note

The current 290 MB of bundled models is the result of skipping the broken static-INT8 path. When the INT8 fix lands, the same accuracy runs at ~120 MB. The shipped FP32 version is correct and fast — but we're paying ~170 MB of phone storage for the bug we didn't chase to root cause yet.
