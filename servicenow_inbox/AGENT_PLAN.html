01Mission — the north star

Close the gap between what the records say is in the rack and what's actually in the rack — automatically, continuously, and for free — by making every incident handled a moment where the data center re-verifies itself.

That single sentence is the design contract. Everything else — parsing, reasoning chains, work notes, batching, anomaly cards — is a side effect of that mission. If a feature doesn't push the gap between records and reality smaller, it doesn't ship.

Why this framing matters. Most "incident triage" agents stop at "the ticket was handled." This one's primary output is "one more cell in the CMDB×reality grid is now confirmed or flagged." Triage is the side effect. Reconciliation is the product.

02The problem this dissolves

A data center has two parallel universes that are always drifting apart. Nobody has 200 hours a month to walk every rack and audit, so drift compounds silently until it causes an outage, an audit failure, or a 3am scramble through stale information.

The world of recordsThe world of facts
CMDB says SW-CORE-01 is at U22 A server is actually at U22; the switch moved in 2023
Ticket says "port 12 on core switch is down" The tech finds port 12 empty — port 14 is the one cabled
Asset DB has a serial number for that PSU The PSU was swapped in a midnight maintenance; nobody updated the row
The runbook says "check the patch panel mapping" Patch panel labels haven't matched reality for two years
Monitoring alert pointed to a device name That name was renamed during a refresh; the alert template never caught up

Reconciling these two worlds is everyone's job and therefore nobody's job. The agent's mission is to turn every incident into a quiet audit — each ticket flowing through it re-grounds one tiny slice of the data center against physical truth, and any drift gets flagged, posted to ServiceNow, and fixed as a normal part of work.

03Stakeholders — who benefits

The tech on the floor

From vague ticket to ranked route

Today: A vague ticket, a building badge, a hunch about which row. 15–20 min of cognitive overhead before they even leave the desk.

With the agent: A ranked route with exact rack + U position, what the last scan saw there, what to bring, what to verify, in their phone — before they stand up.

NetOps lead

From flat queue to triaged plan

Today: A queue of unrelated tickets, manually prioritized.

With the agent: A queue ranked by real impact, batched by location, with rack drift and recurring-failure clusters surfaced before they become the third outage.

Leadership & audit

From hope to evidence

Today: "We think the CMDB is mostly right."

With the agent: A continuously-updated diff between CMDB and reality, with timestamps and evidence per asset — compliance gets a defensible record for free.

The night-shift junior

Tribal knowledge becomes transferable

Today: "Ask Senior X, they remember how this rack is laid out."

With the agent: Senior X's intuition is encoded as patterns the agent applies and grows — juniors get expert-quality calls on night one.

The data center itself

From mute to self-documenting

Today: The building can't tell you what it actually looks like.

With the agent: A live, machine-readable map of its physical state, updated every time someone touches it.

Finance & procurement

From SaaS bills to laptop costs

Today: "AI features" mean per-token bills, vendor risk, lock-in, renewal cycles.

With the agent: Zero SaaS spend. Zero per-call cost. A free Python script that does the work of a paid platform.

04Five non-negotiable principles

  1. Every incident is an opportunity to re-verify reality. The agent's primary output is "one more cell in the CMDB×reality grid confirmed or flagged." Triage is the side effect.
  2. The agent never bluffs. When signals conflict or are weak, it routes to a human. Confidence is reported, not faked. Trust is built one calibrated answer at a time.
  3. Tribal knowledge gets captured, not paraphrased. When a senior person says "oh, that's always the PSU," that becomes a pattern in the dictionary — with their reasoning attached — not a vague summary generated by a model.
  4. The data center is source of truth, the records follow. When CMDB and a scan disagree (within scan-confidence bounds), the scan wins. The agent's job is to make the records catch up to reality, not the other way around.
  5. Free, forever, on a laptop. Any feature that requires a paid API, hosted model, or ongoing subscription is out of scope by definition. The constraint is part of the mission, not a limitation of it.

05The closed-loop data flywheel

The agent isn't a script that runs against tickets — it's a feedback loop where every output becomes input for the next decision. The flywheel is the reason it gets smarter over time without ever calling a model.

┌────────────────────────────────────────────────────────────┐ │ │ v │ ┌─────────────────┐ ┌─────────────────┐ ┌──────────────┴────┐ │ ServiceNow │ ──────► │ AGENT │ ──────► │ Tech walks rack │ │ incidents │ poll │ extract+reason │ route │ scans + fixes │ └─────────────────┘ └─────────────────┘ └──────────────┬────┘ ▲ ▲ │ │ │ │ │ │ │ work-note │ │ work-note │ scan output │ (analysis) │ ▼ ▼ │ ┌─────────────────┐ ┌──────────────────┐ │ │ CMDB lookup │ ◄──────────── │ device_unit_map │ │ │ (read only) │ ground │ (rack scans) │ │ └─────────────────┘ truth └──────────────────┘ │ ▲ │ │ │ │ └──── outcome capture ─────┴─── scan-history diff ──────────────┘ (ticket resolved? in what time? was the diagnosis right?)

The non-obvious arrows are the bottom ones — outcome capture and scan-history diff. They're what turn a one-shot parser into a system that learns.

06Architecture — modules & data flow

The current code is a single poll.py. The bigger picture is a small set of cooperating modules — each with one job, each unit-testable, each replaceable.

servicenow_inbox/ ├── poll.py ← orchestrator: every N sec, runs the pipeline ├── ingest.py ← verifier: CMDB vs scan, per-incident ├── agent/ │ ├── __init__.py │ ├── extract.py ← Phase 1: messy text → structured dict │ ├── patterns.py ← keyword dictionary (grows from unmatched.log) │ ├── word_numbers.py ← "third" → 3, "first uplink" → 1 │ ├── fuzzy.py ← rapidfuzz wrapper over CMDB device list │ ├── reasoning.py ← Phase 2: build reasoning chain step-by-step │ ├── templates.py ← Phase 2: suggested-action templates │ ├── rank.py ← Phase 3: score + batch + anomaly detect │ ├── post.py ← Phase 4: work-note dedup + auto-post │ ├── outcomes.py ← captures resolution data, tunes weights │ ├── sweeps.py ← proactive: stale-rack scans, drift hunts │ └── confidence.py ← multi-signal scoring + calibration ├── state/ │ ├── posted.json ← Phase 4: work-note dedup state │ ├── unmatched.log ← extraction misses, weekly human review │ ├── outcomes.jsonl ← every (prediction, resolution) pair │ └── pattern_weights.json ← learned weights, snapshotted weekly └── AGENT_PLAN.html ← this file

Why this split. Each file is small (<200 LOC), single-purpose, and trivially mockable. extract.py is the only module that touches raw text; fuzzy.py is the only one that calls rapidfuzz; post.py is the only one that writes to SN. If you ever want to swap fuzzy matching for a different algorithm, you change one file.

07Where the intelligence comes from

Six deterministic signals. Each cheap, each auditable, each combined to produce something that looks like AI to a leadership demo — and is AI by any honest 1980s-AI textbook definition.

#SignalWhat it doesData source
1Pattern dictionaryMaps failure keywords to buckets: down, slow, no_link, cable, replace, unreachable, blinking_amber, flappingHand-curated dict in patterns.py, grows from unmatched.log
2Fuzzy CMDB match"core switch 1" → SW-CORE-01 via Levenshtein against actual CMDB device namesrapidfuzz + CMDB list already fetched in poll.py
3Word-to-number"the third port" → 3, "second uplink" → 2Tiny static dict, no library
4Scan-history diffCompare current scan against last scan for the same rack — flag ports that flipped state, devices that movedoutputs/<rackId>/device_unit_map.json history
5Time-window correlationN incidents in <1h pointing to the same rack ⇒ likely shared root cause; surface as a single batchactive_tickets.json
6Class-mismatch detectionCMDB says U10 is a switch; last scan saw a server there ⇒ rack driftCMDB lookup vs scan output (both already in poll.py)
Confidence scoring — the honesty layer. Each signal contributes 0–1. Multi-signal agreement raises confidence; conflicting signals lower it; a single weak signal caps at 0.4 ⇒ ticket goes to a "needs human review" bucket. The agent says "I don't know" when it doesn't — that's what makes it trustable over time.

09Build phases — tactical roadmap

Each phase ships independently behind a PR. None blocks the next, but they layer well in order.

1
Smarter extraction — rules + fuzzy match
~1 day

New file: agent/extract.py exposing extract_incident(text, cmdb_devices) → dict.

Returns

{
  "failure_mode": "port_down | cable_swap | device_unreachable | hardware_replace | config_change | other",
  "affected_device": "SW-CORE-01",
  "affected_port": 12,
  "affected_role": "uplink | access | management | unknown",
  "urgency_signal": "user_reported | monitoring_alert | scheduled",
  "one_line_summary": "WEB-01 lost link, port 12 on access switch suspected",
  "confidence": 0.78,
  "signals_used": ["keyword:down", "fuzzy_match:SW-CORE-01", "word_num:third->3"]
}

How it works

  1. Tokenize short_description + description.
  2. Pattern dictionary hit → failure_mode + role.
  3. Word-to-number expansion → resolve "third", "first", "Gi1/0/12".
  4. Fuzzy device match with rapidfuzz.process.extractOne(text, cmdb_names, score_cutoff=85).
  5. One-line summary = template fill, not generation: f"{device} {mode_human}, port {port} suspected".
  6. Confidence = min(1.0, sum(signal_weights)).

Wiring

Call from poll.py:ticket_record() after the existing regex pass. If the agent agrees with regex → confidence boost. If they disagree → keep the higher-confidence one and log both to state/unmatched.log.

Definition of done. "the third uplink on the access switch in row 4 went dark around 9am" → regex returns null, agent returns {device: SW-ACC-01, port: 3, role: uplink, urgency: user_reported, confidence ≥ 0.6}.

4
Auto-post work notes — closes the loop
~0.5 day

Without anyone clicking anything, every actionable incident gets a work note posted back to ServiceNow with the agent's analysis. This is where "automation" becomes literal.

Work note template

RackTrack Agent Analysis — auto-generated

Incident: WEB-01 unreachable, port 12 suspected
Confidence: 0.82

Reasoning:
  1. Parsed text: failure_mode=port_down, device=SW-CORE-01, port=12
  2. CMDB: SW-CORE-01 lives in RACK-04, port 12 = Gi1/0/12
  3. Last rack scan (2026-05-18): port 12 was EMPTY
  4. Drift detected: CMDB expects port 12 CONNECTED

Suggested action:
  Walk to RACK-04 (U10), verify cable on port 12, check patch panel PP-01.

Related incidents in same rack: INC0010003, INC0010005

Guards (so it doesn't spam SN)

  • Track posted incidents in state/posted.json — {incident_number: analysis_hash}.
  • Re-post only if the analysis hash changes (new scan data changed the conclusion).
  • Hard cap: 1 post per incident per 24h.
  • Confidence floor: post only if confidence ≥ 0.5. Below that → log only.
Definition of done. A fresh SN incident, within one poll cycle, gets a work note posted automatically with the full reasoning chain. Re-running with no changes is a no-op.
5
Outcome capture — the learning loop
~1 day

Until now the agent has only been an emitter. Phase 5 makes it a learner: every resolved ticket becomes a training signal — without training a model. Pattern weights, fuzzy thresholds, and template selections get tuned from real outcomes.

What gets captured

  • Resolution code from SN (resolved, cancelled, duplicate, false-positive)
  • Time-to-resolve
  • Resolution notes text (parsed for "agent was right" / "agent was wrong" signals)
  • Whether the suggested action matched the actual fix

What gets tuned

  • Pattern weights — patterns that consistently lead to right calls get +, wrong calls get −
  • Fuzzy threshold per site — sites with messy naming may lower the cutoff over time
  • Template ranking — templates that mapped to actual fixes win in future tie-breaks
  • Confidence calibration — if "0.8" calls are right 60% of the time, the scale is recalibrated

All tuning is logged, snapshotted weekly, and human-reviewable. Bad updates can be rolled back. No silent drift.

Definition of done. Pattern weights are persisted in state/pattern_weights.json, updated weekly from outcomes, and the agent's accuracy improves measurably across two consecutive months.
6
Proactive sweeps — between incidents
~1 day

Until now the agent only acts when a ticket arrives. Phase 6 makes it look at the data center even when nothing is on fire.

Sweeps

  • Stale-rack sweep. "We haven't scanned RACK-09 in 96 days." Surface as proactive item; suggest next-visit candidate.
  • Drift hunt. Cross every recent scan with current CMDB, surface mismatches that haven't been ticketed.
  • Pattern accumulation review. Weekly digest of unmatched-log entries → suggested new patterns for human approval.
  • Recurring-failure detection. "This rack had 6 port-down tickets in 30 days, 4 on Thursdays." Predict and surface.
  • Cable-age cohort. Cables installed before date X are showing higher failure rates → suggest preventive swap.
Definition of done. The app's "today" view shows at least one proactive card per week that wasn't tied to an incident.

10Proactive behaviors — what the agent does between incidents

An incident-driven agent is just a smarter inbox. The mission demands the agent works between incidents too — turning idle time into reconciliation time.

Stale-rack alerts

Every rack has a "last verified" timestamp from the last scan. After a threshold (60 days, configurable) the agent surfaces the rack as a sweep candidate — "include this in next physical visit."

Drift digest

Weekly: cross every recent scan with CMDB, surface every mismatch that hasn't been ticketed. Email/queue items the NetOps lead can triage in 5 minutes.

Pattern accumulation

Weekly: the unmatched.log gets summarized into "phrases the agent saw and couldn't parse." Senior NetOps reviews and approves new patterns in batches.

Recurring-failure detection

Slow analytics over the resolved-incident log: which racks have above-baseline failure rates, which device families, which times of day. Surface as predictive heat-map.

11Fleet-wide intelligence

Single-incident reasoning is table stakes. The real differentiator is what the agent sees when it looks across many incidents, many racks, many days:

  • Shared-root-cause batches. Three port-down tickets in RACK-04 within 22 minutes ⇒ one PSU fault, not three cable problems.
  • Cross-rack patterns. Every "device unreachable" ticket in row 4 happens at 2am Thursdays ⇒ scheduled job is misconfigured, not hardware.
  • Vendor / firmware cohorts. Failures concentrate on a specific firmware version ⇒ the fleet view surfaces it before a vendor advisory does.
  • Geography of drift. Site A's CMDB-accuracy is 96%; site B's is 71% — resource allocation decision lives in the gap.
  • Tech routing optimization. Given today's queue + the building map, what's the minimum-walk visit order? (Trivial TSP on a small graph.)

12The learning system (still no LLM)

This is the part most teams miss. You can build a sharp deterministic agent that learns — the trick is to learn weights and thresholds, not representations.

What gets learnedHowWhere it lives
Pattern weightsPatterns whose tickets resolved as predicted get +; mis-resolved get −. Capped, decayed, snapshotted.state/pattern_weights.json
Fuzzy thresholds (per site)If a site has many "low-confidence but right" cases, lower the cutoff for that site only.state/site_config.json
Confidence calibrationBrier-score-style: if 0.8 predictions are right 60% of the time, the curve gets bent.state/calibration.json
Template rankingTemplates that matched the actual resolution beat templates that didn't.state/template_stats.json
New patterns (human-approved)Unmatched phrases → weekly review → human says "yes, add it" → PR.agent/patterns.py (code, not state)
What is not learned. No embeddings. No neural net. No "the agent figured out the rules itself." Every learned weight is human-inspectable, human-rollback-able, and tied to a real outcome. That is the difference between learning and drifting.

13Operational surface

Observability of the agent itself

  • Per-poll: tickets seen, tickets actionable, tickets posted, avg confidence
  • Per-week: extraction accuracy (post-resolution backtest), pattern hit rate, fuzzy miss rate
  • Per-month: drift caught, drift remaining, CMDB accuracy by site

Kill switches

  • --no-post — observe only, never write to SN
  • --site=<X> — restrict to one site for safe rollout
  • --max-posts-per-hour — rate limiter
  • Single env var RACKTRACK_AGENT_DISABLED=1 hard-stops auto-posting

Audit trail

  • Every posted work note is hashed and stored locally
  • Every prediction + outcome pair is appended to outcomes.jsonl
  • Every pattern-weight update is logged with the outcome that caused it
  • Easy to answer "why did the agent say that, on date X" months later

Failure modes & recovery

  • SN unreachable → agent keeps running, queues posts, retries next cycle
  • Scan corrupted → ticket flagged "no recent scan", no auto-post
  • Pattern weights look bad → rollback to last weekly snapshot is one file copy
  • Poller crashes → Task Scheduler restarts on failure

15Demo script — 90 seconds for leadership

  1. Setup shown: terminal running the poller, app open on a phone (or browser).
  2. In ServiceNow: show a fresh incident — no work notes yet.
  3. Wait ~60 seconds. Refresh the incident. Work note appears with the agent's full reasoning. "Nobody clicked anything."
  4. In the app: open the Tickets tab. Show the ranked queue with the top item flagged "BATCH: 3 incidents in RACK-04 — possible shared root cause."
  5. Tap the top ticket. Show the "Why this rack?" reasoning panel — 5 steps, evidence per step, confidence pills.
  6. Show an anomaly card: "Rack drift detected — CMDB and last scan disagree on position U22."
  7. Show the "needs human review" bucket: one low-confidence ticket the agent honestly admits it doesn't know. "It doesn't bluff."
  8. Closer: "Zero API cost. Runs on a laptop. Catches things humans miss. Posts back to ServiceNow without being asked. Gets sharper every week."

16Non-goals — what this is not

  • Not an LLM wrapper. Ever. If a feature can't be done deterministically, it doesn't ship.
  • Not a replacement for techs. Force multiplier, not substitute — humans still walk the floor and make the call.
  • Not a CMDB writer (yet). Surfaces drift; doesn't auto-correct. Auto-write needs governance, not a v1 feature.
  • Not an incident closer. Humans close incidents. Agent suggests, evidences, prepares — never resolves.
  • Not a monitoring system. Reacts to incidents that monitoring or humans raise; doesn't watch network health itself.
  • Not a port-as-CI modeler in ServiceNow. Already decided.
  • Not multi-tenant SaaS. One install per data center; the install is the product.
