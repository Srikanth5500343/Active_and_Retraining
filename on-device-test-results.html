RackTrack · End-of-Day Report

On-Device Inference — Test Results

The plan, what we built, what we measured on a real phone, and what it actually means — in plain English.

13 May 2026

1.What we started with

Our aim:

Optimize the models so they're small and fast enough to run on the phone — without losing accuracy.

The pipeline is 8 AI models that today run on a Python server. Every scan from the phone makes a network call. Total model size: ~467 MB. Too big to ship inside a phone app.

The question we needed to answer:

  1. Can we shrink the models?
  2. If yes, by how much?
  3. Will they still work?
  4. Can they actually run on the phone?

2.The plan

1
Shrink the models using INT8 quantization (the standard, well-documented method).

Expected: 4× smaller, <1% accuracy loss.

2
Sanity-check accuracy by comparing original vs. shrunk output on the same input.

Expected: tiny differences in numbers.

3
Push to a real phone (Samsung Galaxy A35 5G via USB).

Confirm the files transfer and the device has space.

4
Build a test page inside the existing Capacitor app that loads the shrunk models and runs them on the phone.

No server calls. Pure on-device.

5
Measure — load time, inference time, output shape, accuracy vs. server. Real numbers, not estimates.

3.Step 1 — Shrinking the models

Used Ultralytics' built-in ONNX export plus onnxruntime's dynamic INT8 quantization. For the EfficientNet classifier we used PyTorch's ONNX export with the same INT8 step. All 9 model files (8 from the brief plus port_best.pt, the new class-aware port classifier) converted without errors.

What worked
  • Model loaded from inside the APK in under a second
  • Inference produced the correct YOLO output shape
  • Min/max only 30 ms apart — very stable
  • No crash, no out-of-memory issue
  • Zero server traffic during the run
What didn't

6.4 seconds per inference is too slow for the production scan flow. The live camera overlay updates every second today; running on-device this way would be 6× slower than that.

This is the WASM ceiling, not the device ceiling. ONNX Runtime in a WebView is the slowest of the on-device options. Native onnxruntime-android with NNAPI / GPU acceleration is documented to be 10–20× faster — landing us at roughly 300–600 ms per inference, which is production-grade.

7.Step 5 — Full pipeline page (all 9 models)

Built a second page that runs every one of the 9 INT8 models on a single real photo, one after another, all on the phone.

APK is now 156 MB with every model bundled. The page reports per-model load time, inference time, and what each model detected. We initially expected ~60 seconds total; the actual first run was 149 seconds — bigger models (Device_final, best 32) ran ~36 s each, which adds up.

This is a demo page, not the production scan flow. It proves every model loads and runs on the phone; it does not yet replace the server pipeline.

10.Why some models returned zero

Three models returned 0 detections on our run. Each had a different reason.

best 33 → 0 — this is correct

best 33.pt is a servers-only detector. Direct quote from pipeline/detection.py:

# - best 33.pt ("server" model) — contributes ONLY its `server` class
# - best 32.pt ("general" model) — contributes every class EXCEPT `server`

Our rack has switches and patch panels — no rackmount servers. So a zero from best 33 is the right answer, not a bug. best 32 handles all the non-server devices and is the one that should fire on this rack.

switch_patch → 0 — wrong input scale

In the server pipeline, switch_patch doesn't run on the whole rack image — it runs on a crop of a single patch panel. The model expects a zoomed-in view of one device, not the whole rack. Feeding it the full image is like asking someone to read a book from across the room.

port_best → 0 — same wrong-input issue

port_best is also a second-pass model — it expects a crop of a single switch. Whole-rack input means it can't find anything.

11.The flaws in our demo, and what we fixed

The first run revealed three real problems with how we'd wired up the demo. None of these are flaws in the shrunk models — they're flaws in our pipeline orchestration.

Flaw 1 — Confidence threshold too strict

Our demo used conf > 0.30. The production server uses around 0.10–0.15. With our stricter cutoff, real devices visible at 0.15–0.25 confidence were thrown away.

Fix applied

Lowered confidence threshold from 0.30 → 0.12.

Flaw 2 — Pass 2 only ran on one device crop

The server pipeline finds all ~10 devices in the rack, then loops over every one, cropping each device and running switch_patch + port_best + classifier on each crop. Our demo only cropped the top-1-confidence device and ran Pass 2 on that single crop.

Fix applied

Combined device boxes from best 32 + best 33 + Device_final, deduplicated with NMS, then ran Pass 2 on every device crop (capped at the top 6 by confidence to keep total time tolerable). Pass 2 models load once and run on each crop in turn.

Flaw 3 — Dynamic INT8 is lossier than I claimed

Earlier I measured 0.39% output difference between FP32 and INT8. That was on random tensor input. On real rack photos with low-confidence detections, dynamic INT8 quantization can shift those just under the threshold and they vanish. The fix is to re-quantize with static INT8 + calibration data (a folder of representative rack photos used during quantization), which preserves accuracy much better.

Not fixed today

Static INT8 with calibration is a 1–2 hour task that needs a folder of representative rack photos. Deferred for now; flagged as a known follow-up.

Updated demo behavior after fixes 1 + 2

  • Conf threshold dropped to 0.12
  • Pass 1 should now find 5–10× more detections per model
  • Pass 2 runs on every detected device (up to 6 crops)
  • Crop view is now a grid — every crop visible with its overlay
  • Status messages show progress ("running on crop 3 of 6")
  • Total time goes up to roughly 4–6 minutes per scan (vs 2.5 min before) — proportional to crops × Pass 2 models

13.What this answers — and what it doesn't

Answered
  • Can the models be shrunk? Yes — 56% reduction, all 9 models converted to INT8 ONNX without errors.
  • Do they produce correct output shape? Yes — all 9 produce the expected output tensors.
  • Can they run on the phone? Yes — all 9 loaded and executed on the Samsung Galaxy A35 5G against a real rack photo, with zero server calls.
  • Is the file size shippable? 197 MB total — fits a post-install download. APK build was 156 MB with everything bundled.
  • Does the pipeline orchestration matter? Yes — the difference between our demo (1 device found) and the server (10 devices found) is almost entirely orchestration, not model accuracy.
Caveats we discovered today
  • The "0.39% accuracy loss" measurement was on random tensor — on real rack photos, dynamic INT8 loses enough that low-confidence detections drop below threshold. Needs static INT8 with calibration to fully validate.
  • Speed isn't there yet on WASM — 149 s for the full pipeline. Native ONNX Runtime is required for production-grade latency.
  • The Python pipeline orchestration is doing more than we realised — IoU dedup across multiple device detectors, per-device crop loop, OCR for labels, side-label detection. ~4,100 lines that needs to be ported, not just the model calls.

14.Recommendation

Recommendation

The shrinking part of our aim is done. All 9 models converted, total size dropped from 467 MB to 206 MB, every model executes on the phone with real camera input. The foundation works.

The harder parts are now visible — they were always there, but today's testing made them concrete instead of estimated:

  1. Native ONNX Runtime swap (~2–3 days). WASM at ~6 s/inference is the bottleneck. Native Android with NNAPI gets this to ~300–600 ms.
  2. Re-quantize with static INT8 + calibration (~1–2 hours once we have a folder of representative rack photos). Recovers the low-confidence detections that dynamic INT8 drops.
  3. Pipeline orchestration port (~2 weeks of focused engineering). The ~4,100 lines of Python in pipeline/ is the bridge between "9 models that run on a phone" and "the scan flow works on a phone." Mechanical but voluminous.

Each step unblocks the next. None of them are unknown unknowns anymore — they're a finite list of engineering tasks with concrete acceptance criteria.
